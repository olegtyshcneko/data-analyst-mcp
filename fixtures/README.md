# Test fixtures

Two fixture families, both generated by deterministic scripts in this
directory. The generated CSVs are committed (they are demo / test data,
not build artifacts) and must be byte-identical on every re-run.

| Generator | Output | Seed |
|---|---|---|
| `_build_messy.py` | `messy.csv` | `20260511` (single `random.Random`) |
| `generate_synthetic_crm.py` | `synthetic_crm/{accounts,contacts,opportunities}.csv` | `20260511` (Python `random`, NumPy, and `Faker.seed`) |

Run either generator directly to regenerate:

```bash
uv run python fixtures/_build_messy.py
uv run python fixtures/generate_synthetic_crm.py
```

---

## `messy.csv` — the hostile CSV

5000 data rows + 1 header row. 12 columns. Deliberately broken in nine
distinct ways so that the downstream profilers, importers, and EDA tools
have a concrete adversary.

| # | Failure mode / property | Column(s) | Exact count / shape |
|---|---|---|---|
| 1 | UTF-8 BOM prefixes the file | (file-level) | First 3 bytes = `EF BB BF` |
| 2 | Trailing whitespace in column headers | `revenue ` and `last_login ` | Exactly 2 of 12 |
| 3 | Three mixed date formats | `signup_date` | ISO, `DD/MM/YYYY`, `Mon DD, YYYY` — each appears >0 times |
| 4 | Two distinct "missing" representations | `revenue` | 200 rows = `"N/A"` (4%), 100 rows = empty (2%) |
| 5 | Comma inside a quoted field | `notes` | Exactly 1 row; CSV-quoted so DuckDB parses cleanly |
| 6 | Two byte-identical duplicate rows | (whole row) | `duplicated().sum() == 1` |
| 7 | Mostly-null email column | `email` | Exactly 3900 empty cells (78%) |
| 8 | IQR outliers + data-entry errors | `score` | 20 IQR outliers, 5 of which are wildly large (`>10000`) |
| 9 | Case-inconsistent categorical | `country` | 95% in `{PL,US,DE,UA}`, 5% in `{poland, Poland, POL, "pl "}` |

### Columns

`customer_id, signup_date, country, revenue , score, email, notes, plan, age, last_login , channel, is_active`

(The two columns with trailing whitespace appear above with their literal
header strings. `revenue ` and `last_login ` are intentional — issue #2.)

### Deterministic construction notes

- The 20 outlier positions for issue #8 are sampled *before* score
  generation so inliers (clamped Gaussian, mean 70, sd 8) fill only
  non-outlier rows. Empirical fences land at roughly `[49.45, 90.65]` so
  every planted outlier (range 20–35, 105–130, plus 99999-class values)
  is comfortably outside.
- The two duplicate rows for issue #6 are picked from rows that are
  *not* members of any other issue's index set. This way the duplicate
  overwrite leaves all other counts undisturbed.

## `synthetic_crm/` — three-table CRM dataset

Realistic-shape CRM with full referential integrity for joins, ANOVA
across opportunity stages, logistic regression on `won ~ amount +
C(industry)`, chi-square on industry × won, and so on.

| File | Rows + header | Columns |
|---|---|---|
| `accounts.csv` | 2000 + 1 | `account_id, account_name, industry, country, employees, arr, created_at, churned` |
| `contacts.csv` | 20000 + 1 | `contact_id, account_id, first_name, last_name, email, title, is_primary` |
| `opportunities.csv` | 8000 + 1 | `opp_id, account_id, stage, amount, created_at, closed_at, won` |

### Properties exercised

| Property | Where | Detail |
|---|---|---|
| Log-normal ARR | `accounts.arr` | `round(exp(N(11, 1.2)), 2)` — describe_column should report strong positive skew |
| Industries | `accounts.industry` | 8 categories: Software, Manufacturing, Healthcare, Retail, Finance, Education, Logistics, Energy |
| Countries | `accounts.country` | 12 categories |
| Churn flag | `accounts.churned` | ~10% (verifier asserts within [6%, 14%]). **Extension to the spec** — see below. |
| FK integrity | `contacts.account_id`, `opportunities.account_id` | Zero orphans (verifier-enforced) |
| Every account has ≥1 contact | `contacts` | Enforced at generation |
| ≥70% of accounts have ≥3 contacts | `contacts` | Generator boosts ~80% of accounts to ≥3 |
| Exactly one primary contact per account | `contacts.is_primary` | Enforced at generation |
| Opportunity stage enum | `opportunities.stage` | `Prospecting, Qualification, Proposal, Negotiation, Closed Won, Closed Lost` |
| ~70% closed | `opportunities.closed_at` | Verifier asserts closed-fraction ∈ [0.60, 0.80] |
| Win rate ~28% (of closed) | `opportunities.won` | Verifier asserts ∈ [0.20, 0.36]; open opps have NULL `won` |
| Amount ↔ ARR correlation | join `opportunities ⋈ accounts` | `amount = 0.05 * arr * exp(N(0, 0.5))` — Pearson ≥ 0.25 (verifier-enforced) |

### Deliberate extensions to the spec

- **`accounts.churned` column.** Spec §7 specifies "churn flag on ~10%
  of accounts" without naming the column. We added a `churned` boolean
  column on `accounts.csv`. Recording this here so anyone reading the
  spec doesn't think it drifted by accident.

### Determinism notes

- Three seed sources are reset on every run: `random.seed(20260511)`,
  `np.random.seed(20260511)`, and `Faker.seed(20260511)`. The generator
  also takes a fresh `random.Random(20260511)` and
  `np.random.default_rng(20260511)` to keep state isolated.
- `account_id` uses a UUID-shaped hex string drawn from the seeded
  `random.Random` (not `uuid.uuid4`, which is OS-entropy-backed and
  therefore non-deterministic).
- CSVs are written with explicit `lineterminator="\n"` and
  `QUOTE_MINIMAL` so output is byte-stable across platforms.
