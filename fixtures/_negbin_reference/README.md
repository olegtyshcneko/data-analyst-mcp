# Negative binomial reference suite

Three small synthetic count datasets (`ref1.csv`, `ref2.csv`, `ref3.csv`) plus an `expected.json` sidecar pinning the negative binomial (NB2) fit output for each.

These reference values were generated by the **same** `statsmodels.formula.api.negativebinomial(...).fit(disp=False)` call that `fit_model(kind="negbin")` uses internally — they are pinned snapshots of the current statsmodels implementation rather than independently-derived ground truth.

The regression test `test_fit_model_negbin_matches_reference_suite` (in `tests/test_models.py`) asserts the live tool reproduces these numbers to ≤1e-3. It catches:

- accidental parameterization changes (NB1 vs NB2),
- coefficient/SE plumbing regressions,
- silent statsmodels-version drift.

## TODO — swap in R's `MASS::glm.nb` reference

The proposal (`docs/proposals/negbin_kind.md` Acceptance section) calls for ≤1e-3 agreement with R's `MASS::glm.nb`. R is not installed in this Phase 2 environment, so we pin to statsmodels' own output to get *some* regression coverage now and defer the R cross-validation to a follow-up.

To upgrade: install R + the MASS package, run `MASS::glm.nb(<formula>, data=read.csv("refN.csv"))` for each fixture, replace the coefficient/SE/alpha values in `expected.json` with R's output, and the existing test will tighten automatically.

## Regenerating

If statsmodels' NB MLE changes (rare, but possible across major releases), regenerate via:

```bash
uv run python -c "
import json, os, numpy as np, pandas as pd
import statsmodels.formula.api as smf
# ...see git history for the generator snippet
"
```

The generator uses fixed seeds (`101`, `202`, `303`) so the CSVs themselves are stable.
